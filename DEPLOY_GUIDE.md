# 稿件硬伤猎手 — 部署与配置指南

## 第一部分：云服务器部署（约30分钟）

### 1.1 购买云服务器

推荐配置：

| 项目 | 推荐 |
|------|------|
| 平台 | 腾讯云 / 阿里云 |
| 规格 | 2核2G（轻量应用服务器最便宜） |
| 系统 | Ubuntu 22.04 |
| 带宽 | 3Mbps 够用 |
| 预算 | 比赛期间约 30-50 元/月 |

购买后你会得到一个 **公网 IP**（例如 `43.135.xxx.xxx`），记下来。

### 1.2 登录服务器

用 SSH 登录（Windows 可用 PowerShell 或 MobaXterm）：

```bash
ssh root@你的公网IP
```

输入购买时设置的密码。

### 1.3 安装 Python 环境

```bash
# 更新系统
apt update && apt upgrade -y

# 安装 Python（Ubuntu 22.04 自带 3.10）
apt install -y python3 python3-pip python3-venv

# 验证
python3 --version
# 应显示 Python 3.10.x
```

### 1.4 上传项目文件

在你的 **本地电脑** 执行（把项目上传到服务器）：

```bash
# 方法1：用 scp 上传整个项目
scp -r "h:\00 Studio\0418_risk_checker - Copy" root@你的公网IP:/root/fact_checker

# 方法2：如果 scp 不方便，用 MobaXterm 的文件拖拽功能
```

需要上传的文件（只需要这 5 个）：

```
config.py
fact_engine.py
schemas.py
api_server.py
requirements.txt
start.sh
stop.sh
```

### 1.5 安装依赖 & 启动

```bash
# 进入项目目录
cd /root/fact_checker

# 安装依赖
pip3 install -r requirements.txt

# 启动服务
bash start.sh

# 验证是否启动成功
curl http://localhost:8000/health
# 应返回: {"status":"ok","version":"2.0"}
```

### 1.6 开放防火墙端口

**腾讯云控制台操作：**
1. 登录腾讯云控制台 → 轻量应用服务器
2. 点击你的服务器 → 防火墙
3. 添加规则：
   - 协议：TCP
   - 端口：8000
   - 来源：0.0.0.0/0（允许所有 IP 访问）
4. 确认

**阿里云控制台操作：**
1. 登录阿里云控制台 → 云服务器 ECS
2. 安全组 → 配置规则
3. 入方向添加：
   - 协议：TCP
   - 端口范围：8000/8000
   - 授权对象：0.0.0.0/0

### 1.7 验证公网访问

在你的 **本地电脑浏览器** 打开：

```
http://你的公网IP:8000/health
```

看到 `{"status":"ok","version":"2.0"}` 就说明部署成功。

也可以在本地用 curl 做一次完整测试：

```bash
curl -X POST http://你的公网IP:8000/check/facts \
  -H "Content-Type: application/json" \
  -d '{"content": "2023年北京市GDP达到43760亿元，同比增长5.2%。"}'
```

## 第二部分：初芯平台 Chatflow 配置

### 2.1 创建应用

1. 登录初芯平台
2. 点击 **"创建应用"**
3. 选择 **"Chatflow"** 类型
4. 应用名称填：`稿件硬伤猎手`
5. 描述填：`新闻稿件事实核查助手——基于原子事实分解与联网搜索验证`

### 2.2 编排 Chatflow 节点

进入 Chatflow 编辑器后，你需要搭建以下流程：

```
[开始] → [问题分类] → [HTTP请求] → [LLM格式化] → [结束/回复]
```

#### 节点 1：开始节点

默认已有，不需要改。用户的输入会自动传入。

#### 节点 2：问题分类节点（可选，建议加）

作用：判断用户是在提交稿件还是在追问。

配置：
- 添加一个 **"问题分类"** 节点
- 分类条件：
  - **分类1**：`用户提交了一篇新闻稿件要求审查`
    → 连接到 HTTP 请求节点
  - **分类2**：`用户在追问或询问上次审查结果的细节`
    → 连接到一个直接回答的 LLM 节点

如果觉得复杂，可以先跳过这个节点，直接把开始节点连到 HTTP 请求。

#### 节点 3：HTTP 请求节点（核心）

这是连接你后端 API 的关键节点。

**配置如下：**

| 配置项 | 值 |
|--------|-----|
| 请求方法 | POST |
| URL | `http://你的公网IP:8000/check/facts` |
| Headers | `Content-Type: application/json` |
| Body 类型 | JSON |
| Body 内容 | 见下方 |

**Body 配置：**

```json
{
  "content": "{{#sys.query#}}"
}
```

说明：`{{#sys.query#}}` 是 Dify 的变量，代表用户当前输入的内容。

> 注意：不同版本的 Dify 变量语法可能略有不同。
> 如果 `{{#sys.query#}}` 不生效，尝试 `{{#start.content#}}` 或在开始节点中查看输出变量名称。

**超时设置：** 建议设为 120 秒（核查需要多次搜索，耗时较长）

#### 节点 4：LLM 格式化节点

作用：把 API 返回的 JSON 转成用户能看懂的报告。

**模型选择：** 平台上可用的模型（Qwen 等）

**System Prompt：**

```
你是"稿件硬伤猎手"，一个专业的新闻稿件核查助手。
你的职责是将审查引擎返回的结构化结果整理为清晰的审查报告。

报告格式要求：
1. 开头用一句话给出整体风险评级（🔴高危 / 🟡存疑 / 🟢通过）
2. 按风险等级从高到低排列各项发现
3. 每条发现包含：原文摘录 → 问题描述 → 核查依据 → 建议修改
4. 通过项简要列出即可，不需要展开
5. 结尾给出"建议处理优先级"

语气要求：专业、克制、不夸大风险，但对硬伤零容忍。

注意：
- 如果 evidence_urls 存在，在核查依据中自然地提及来源
- "错误"项用 ❌ 标记
- "存疑"项用 ⚠️ 标记
- "通过"项用 ✅ 标记
```

**User Prompt：**

```
以下是审查引擎返回的核查结果（JSON格式），请整理为审查报告：

{{#HTTP请求节点的输出变量#}}
```

> 注意：`{{#HTTP请求节点的输出变量#}}` 需要替换为实际变量名。
> 在 Dify 中，你可以点击输入框旁的 `{x}` 按钮，选择上一个 HTTP 请求节点的 `body` 输出。

#### 节点 5：结束/回复节点

将 LLM 格式化节点的输出连接到 **"回复"** 节点即可。

### 2.3 完整节点连线示意

```
┌────────┐    ┌──────────┐    ┌──────────────┐    ┌────────────┐    ┌──────┐
│  开始  │ →  │ 问题分类  │ →  │  HTTP请求     │ →  │ LLM格式化  │ →  │ 回复  │
│        │    │(可选)     │    │  POST /check  │    │ JSON→报告  │    │      │
└────────┘    │          │    │  /facts       │    └────────────┘    └──────┘
              │  追问 ─────────→ LLM直接回答 ──────────────────────→ 回复
              └──────────┘
```

### 2.4 测试

1. 点击右上角 **"发布"**
2. 进入 **"预览"** 或 **"运行"**
3. 粘贴一段测试稿件：

```
2024年3月15日，北京市副市长张伟在清华大学出席科技创新与城市发展论坛。
张伟指出，2023年北京市GDP达到43760亿元，同比增长5.2%。
论坛期间发布的报告显示，北京市2023年研发投入达到2800亿元，占GDP比重为6.4%。
```

4. 等待 30-60 秒，应该能看到格式化好的审查报告

### 2.5 常见问题

**Q: HTTP 请求超时了？**
A: 把超时时间改大到 180 秒。核查引擎需要多次调用搜索 API，比较耗时。

**Q: 返回结果是空的？**
A: 先在浏览器访问 `http://你的IP:8000/health` 确认服务在运行。

**Q: 变量引用不对？**
A: 在 Dify 编辑器中，点击输入框旁的 `{x}` 图标，手动选择上游节点的输出变量，不要手打变量名。

**Q: 服务器重启后服务没了？**
A: SSH 登录后重新执行 `cd /root/fact_checker && bash start.sh`。
